For example, in Wave 1, the visualiser detected 650 categorical columns, 150 (\texttt{float64}) columns, and 9 object columns, while the harmonised dataset contained 304 categorical columns, 148 float32 columns, and several integer columns. Age distributions confirmed that all participants were $50+$ across waves, and missing value summaries indicated substantial variation (e.g., mean missing values ranging from $4.7\%$ in Wave 4 to $35\%$ in the harmonised dataset, with some variables exceeding $77\%$ missing).

Despite this, the exploratory visualiser was limited because the raw data were not yet cleaned, many outputs were incorrect or nonsensical due to hybrid formats and inconsistent coding. For instance, inspection revealed that many variables across both the wave specific and harmonised datasets, while appearing numeric at first glance, in fact contained hybrid representations combining numeric prefixes with descriptive text (e.g., ``1.Rarely or none of the time (less than 1 day)'', ``4.All of the time (5--7 days)'', or prefixed labels such as ``0.not coupled'', ``1.coupled''). A naive string stripping approach would therefore produce incorrect numeric values (for example, reducing ``5--7 days'' to \texttt{5}) resulting in silently introducing semantic distortion. Similarly, numeric columns sometimes contained sentinel codes (e.g., -98, -99) or embedded symbols ($`\$', `cm', `\%'$) that skewed visual summaries. 

These insights, revealed through systematic profiling, guided the development of the subsequent cleaning pipeline, ensuring consistent numeric logic representations (e.g., midpoints for ranges, standardised sentinel handling).

\section{Cleaning Engine}
The core challenge in converting heterogeneous variable types across the six datasets (Table~\ref{tab:dataset_shape}) was achieving a single numerical standard suitable for downstream analysis. This required enforcing \emph{numerical homogeneity}, defined here as representing all retained features as consistent numerical values (\texttt{float64}) with a unified missing or non response sentinel ($\texttt{-1.0}$). To address this, the solution was implemented as a two stage system. The first stage performs automated \emph{variable scenario detection}, identifying the structural form of each variable based on observed value patterns using the \texttt{detect\_column\_scenario\_mapping} function (Appendix~\ref{app:variable_mapping_code}). This stage includes additional diagnostic categories used during development to surface unhandled edge cases.

The second stage applies a deterministic \emph{six step numerical cleaning engine} (Appendix~\ref{app:cleaning_pipeline_code}). While nine scenario categories were initially defined during development, three served exclusively as validation fallbacks and were progressively eliminated as detection coverage improved. The final pipeline therefore consists of six active transformation steps, applied sequentially and uniformly across all waves.

\subsection{Stage 1: Variable scenario mapping and validation}
Stage 1 focused on automatically identifying how each variable should be handled during cleaning. An automated detection function in Appendix~\ref{app:variable_mapping_code} was developed to classify variables based on their observed value structure, including the presence of sentinel codes, numeric annotations, ordinal labels, and mixed string numeric formats. While multiple intermediate detection categories were used during development, these served solely as internal routing logic. While all variables passed sequentially through the six step cleaning pipeline, most were fully resolved in the earlier steps (Steps 1--5). The final step (Step 6) was applied only to residual variables that could not be deterministically resolved by earlier transformations.

The additional detection categories (e.g., ``single value categorical'', ``special symbol mixed'', and ``unknown/complex'') were intentionally included as diagnostic fallbacks to surface unhandled edge cases due to the complexity and size of the data. As detection rules were continuously refined and coverage improved, variables initially routed to these fallback categories were progressively absorbed into earlier, well defined scenarios, resulting in zero variables remaining in these categories for the final pipeline execution.

To ensure robustness, the classification output was immediately processed by a validation routine that generated a comprehensive dictionary of feature states before and after cleaning, including all representative sample values for each variable. This allowed direct inspection of how the same variable was represented across waves prior to cleaning, particularly in cases where identical semantic categories appeared in different orders or formats. By comparing these grouped samples before and after each cleaning step, classification errors and inconsistent mappings were identified and corrected prior to execution of subsequent transformations.

The dictionary generated inside the ``show\_grouped\_vars\_for\_scenarios" function was formatted to create an automatic friendly ``.md'' file for rapid, surface level verification, which honed the final mapping dictionary in (Appendix~\ref{app:md_audit}) for all features across all waves. This process of using automated detection followed by visual, manual validation was essential for correcting errors in classification logic before the subsequent scenario transformations were executed in the pipeline.

\subsection{Stage 2: Deterministic six step cleaning sequence}
The nine input scenarios were processed sequentially by the \emph{six step cleaning engine}, resolving simple and unambiguous cases first to maximise the proportion of clean numerical features before tackling more complex structures, particularly censored and multi-class scenarios (Figure~\ref{fig:cleaning_flow}). This deterministic sequence applied identical logic across all waves, ensuring consistent data alignment (see Appendix~\ref{app:cleaning_pipeline_code}).

The final cleaning pipeline consisted of six deterministic steps applied uniformly across all \gls{TILDA} waves, with increasingly complex transformations handled in later stages. Steps 4--6 required particular attention due to hybrid, ordinal, or multi-class formats. A summary of each step, including technical details and examples, is provided below.

\begin{enumerate}
    \item \textbf{Empty / Blank Values:} All missing entries, empty strings, and whitespace were converted to the standard non-response sentinel $\texttt{-1.0}$.
    
    \item \textbf{Sentinel Values:} Wave specific non-response codes (e.g., $\texttt{-98}$, $\texttt{-99}$) and textual sentinels (e.g., \texttt{"dk"}, \texttt{"not applicable"}) were mapped to the unified $\texttt{-1.0}$ placeholder.

    \item \textbf{Numeric Transformation:} Numeric like columns containing embedded units, symbols, or hybrid strings (e.g., $\texttt{50+}$, $\texttt{<10}$, $\texttt{1-2.5}$) were parsed using regex and converted to \texttt{float64}. For ranges, midpoints or boundary values were used. For censored or annotated values, appropriate upper/lower bounds were applied. 

    \item \textbf{Censored / Ordinal Variables:} This step addressed variables representing ordinal scales, bracketed numeric ranges, or frequency expressions (e.g., ``2--3 days a week'', ``daily''). Transformations included:
    \begin{itemize}
        \item Converting textual ordinal labels and Likert scale responses to numeric equivalents.  
        \item Mapping calendar based frequency descriptors to numeric weekly equivalents.  
        \item Manual verification for low count or unusual variables, supported by the Markdown audit outputs (Appendix~\ref{app:md_audit}), to ensure semantic meaning was preserved.  
    \end{itemize}

    \item \textbf{Binary Categorical Variables:} Two-level features (e.g., Yes/No, Male/Female) were dynamically encoded using a per variable mapping dictionary:
    \begin{itemize}
        \item Semantic rules were applied e.g., (``no'', ``not'', ``female'' $\to 0.0$) and (``yes'', ``male'' $\to 1.0$).  
        \item Count based detection verified that variables had exactly two non-missing unique values.  
        \item Sentinels were preserved as $\texttt{-1.0}$.  
        \item This approach ensured order invariant encoding across waves, even when original labels or numeric prefixes differed.  
    \end{itemize}

    \item \textbf{Multi-class / Mixed Variables:} All remaining multi-class, mixed type, or string numeric hybrid variables were processed in the final step:
    \begin{itemize}
        \item Lexical parsing extracted numbers, ranges, units, percentages, and textual frequency descriptors into floats.  
        \item Sentinel handling ensured all known non-response codes mapped to $\texttt{-1.0}$.  
        \item Original values were grouped and assigned numeric identifiers in first seen order, non numeric categorical labels were assigned integer codes above the highest observed numeric value to prevent collisions.  
        \item Manual inspection using Markdown audit files verified that semantic equivalence was maintained across waves.  
        \item Step 6 was designed to be \emph{order invariant}, dynamically constructing mapping dictionaries from observed values during each execution rather than reusing mappings from previous waves. This preserved consistent numeric representation regardless of variable ordering or wave specific encodings.  
    \end{itemize}
\end{enumerate}

\subsection{Cross wave semantic verification}
Given the absence of guaranteed category ordering across waves, preserving semantic equivalence during numerical encoding was essential to avoid introducing distortions into downstream \gls{ML} models. Pre and post transformation values for each group were exported to a readable Markdown (\texttt{.md}) file, as shown in Appendix~\ref{app:md_audit}. This representation confirmed that features of known importance for downstream predictive models were preserved and correctly encoded. For example, features such as \texttt{ph001} (self-rated general health), \texttt{ph009} (comparative self-rated health), and \texttt{bh006} (smoking frequency) correspond to items identified by \cite{moguilnerImportanceAgePrediction2021} as among the most influential predictors in mortality models. Retaining these features during the manual filtering thresholds (Chapter~\ref{ch:chapter3}) and encoding them consistently across waves ensured that the cleaned dataset maintains coverage and semantic integrity, preventing distortions that could potentially compromise model performance.

A key challenge in cleaning the five \gls{TILDA} waves was the lack of guaranteed ordering for categorical values of the same variable across different waves. Table~\ref{tab:wave_ordering_example} illustrates this for Wave 4 and Wave 5.

For example, gender variables appeared under different value encodings (\texttt{sex}), with varied category orderings and numeric prefixes across waves. Values such as \texttt{[`Male', `Female']}, \texttt{[`Female', `Male']}, and prefixed forms like \texttt{[``1.male', `2.female']} were all mapped to a consistent numeric representation (\texttt{1.0} for male, \texttt{0.0} for female). This ensured that semantic equivalence, rather than positional ordering or wave specific encoding, governed numerical assignment.

The variable \texttt{ph001} found in Appendix~\ref{app:md_audit} corresponds to the survey question: “Now I would like to ask you some questions about your health. Would you say your health is…”. Responses are ordinal (Poor to Excellent) with non-response categories. 

The grouped Markdown representation reveals two critical points: 
\begin{enumerate}
    \item Semantically identical categories are correctly grouped even under different variable names (\texttt{ph001} vs \texttt{ph009}). 
    \item Numerical encoding is invariant to category ordering, preserving ordinal meaning across waves. Without this approach, positional encoding would assign inconsistent values, potentially distorting performance for downstream models.
\end{enumerate}

\section{Cleaning Validation}

The robustness of the automated cleaning pipeline was evaluated in two stages. First, the internal consistency of cleaned predictors was assessed against established biological relationships. Second, full model replication was performed using a secondary study to confirm predictive utility.

% \subsection{Pairwise correlation validation}
Correlation analysis was conducted on the seven primary predictors identified by \cite{davisLinearRegressionbasedMachine2021} as explaining $\texttt{95\%}$ of the peak (\gls{GSR}) model performance, which is defined in this study as the difference between maximum and usual gait speed. Since the \gls{GSR} outcome itself is not natively available in the public Wave 3 dataset, validation was focused on the directional relationships between the predictors themselves. This internal consistency check ensures that the biological and clinical signals reported in the literature are preserved within the dataset (post cleaning).

Figure~\ref{fig:heatmap} displays the correlation matrix for the selected predictors. These top 7 predictors were used to establish directional relationships for verifying that the numerical parsing pipeline specifically the handling of sentinel values was successful.
